# Infrastructure as Code (IaC)

This directory contains the Infrastructure as Code (IaC) necessary to deploy the Multi-Modal RAG project directly to AWS. Everything here is built using **Terraform** and companion Bash scripts to provide a reproducible, automated, and safe cloud deployment.

## Why Terraform?

Terraform is the industry standard tool for cloud provisioning. Rather than clicking through the AWS Console (which is error-prone and hard to replicate), Terraform allows us to treat our cloud infrastructure like software.

**The Advantages we gain by using Terraform:**
1. **Declarative Configuration:** We write code that describes the *desired end-state* of our infrastructure. Terraform figures out how to make reality match that truth.
2. **State Management:** Terraform keeps an exact ledger (the `.tfstate` file) of what resources have been created. This means it knows exactly what to modify or delete on future updates without accidentally creating duplicates.
3. **Disaster Recovery:** If someone accidentally deletes a Load Balancer in the AWS Console, you simply run `terraform apply` again, and Terraform will automatically recreate exactly what is missing.
4. **Clean Teardowns:** Running `terraform destroy` safely removes all linked resources, preventing orphaned services from racking up surprise AWS billing charges.

## Infrastructure Architecture

Our Terraform code provisions a completely modern, serverless container architecture on AWS. 

Here is a breakdown of the core infrastructure stack that is created automatically:
* **Networking (`vpc`, `security_groups`):** A custom Virtual Private Cloud with dedicated subnets and strict security groups to prevent unauthorized access.
* **Storage (`efs`):** An Elastic File System (EFS) designed to persistently store the Qdrant Vector database data and Ollama AI models across container restarts.
* **Containers (`ecr`, `ecs`, `task_definitions`):** AWS Fargate (Serverless ECS) is used to orchestrated the three required Docker containers (`app`, `qdrant`, and `ollama`), pulling image artifacts from our Elastic Container Registry (ECR).
* **Load Balancing (`alb`):** An Application Load Balancer routes external HTTP traffic securely to the internal API container. 
* **Security & Auth (`iam`, `secrets`):** Granular IAM Roles for the ECS tasks and CI/CD pipelines, plus AWS Secrets Manager implementation for safely passing the OpenAI API key.

## How the Code is Organized (Root vs. Modules)

Our infrastructure uses a **modular** Terraform design. Instead of one massive, unreadable configuration file, we logically split AWS resources into standalone components (modules). 

In both the **Root Level** (`infra/terraform/`) and the **Module Level** (`infra/terraform/modules/xyz/`), you will see the same three core file types. Here is what they do:

### 1. The Root Level (The Orchestrator)
The root directory (`infra/terraform/`) acts as the conductor of the orchestra. It rarely creates resources directly; instead, it calls the modules and passes data between them.
* **`variables.tf` (Root):** Defines the global parameters applied to the whole project (e.g., `project_name`, `aws_region`). You override these locally using `terraform.tfvars`.
* **`main.tf` (Root):** Instantiates the different modules and explicitly controls the dependency flow. For example, it grabs the Subnet IDs generated by the `vpc` module and injects them into the `alb` (Load Balancer) module.
* **`outputs.tf` (Root):** Exposes final, usable artifacts back to the terminal or CI/CD pipeline after a deployment completes (e.g., `alb_public_url`).

### 2. The Module Level (The Workers)
Each folder inside `/modules/` is a self-contained, insulated block of infrastructure.
* **`variables.tf` (Module):** Defines the required inputs for *just that specific module*. It acts exactly like the parameters of a coding function. 
* **`main.tf` (Module):** The worker logic. This is where the actual `resource "aws_..."` blocks exist that provision physical cloud infrastructure on AWS.
* **`outputs.tf` (Module):** Returns dynamically generated AWS data (like newly minted ARNs, Security Group IDs, or VPC IDs) back up to the root `main.tf` so other modules can consume them as inputs.

### 3. Key Core Files (Providers, State, and Vars)
Alongside the main infrastructure orchestration files, there are a few critically important Terraform configuration files that manage the tool itself:
* **`versions.tf` & Providers:** This file tells Terraform exactly which plugins (Providers) it needs to download to interact with external APIs. For this project, it locks the `hashicorp/aws` provider to version `~> 5.0` so our code doesn't break if AWS releases a drastically different API version in the future. 
* **The Terraform State (`.tfstate`):** Terraform's "brain". When you deploy, Terraform creates a state ledger that maps your written code blocks to the actual physical IDs of the AWS resources it created (e.g., this line of code equals AWS VPC ID `vpc-01234`). Because we specifically configured an S3 backend in `versions.tf`, this state data is automatically pushed safely to AWS so multiple team members can deploy seamlessly without overwriting each other.
* **`terraform.tfvars`:** This is your personal configuration file. While `variables.tf` defines *what* inputs the codebase expects, `terraform.tfvars` is where you actually *supply* the real values (like setting `app_memory = 16384`). **This file is `.gitignore`d** because it often contains sensitive AWS configuration meant only for your local machine.

## Directory Structure

```text
infra/
├── TERRAFORM_GUIDE.md       # 📖 The step-by-step tutorial for deploying!
├── setup-backend.sh         # Creates your S3 Bucket for Terraform Remote State
├── setup-github-secrets.sh  # Exports live AWS configurations to GitHub Actions
├── bootstrap-ollama.sh      # A helper script to pull the AI model onto EFS
└── terraform/               # The core Terraform root directory
    ├── main.tf              # The root module connecting all puzzle pieces
    ├── variables.tf         # Configuration input definitions
    ├── outputs.tf           # Extracted artifacts (like the ALB DNS URL)
    ├── terraform.tfvars     # (User-created) Local configuration overrides
    └── modules/             # Segregated Terraform components mapping to AWS 
        ├── alb              
        ├── ecr
        ├── ecs
        ├── efs
        ├── iam
        ├── logs
        ├── secrets
        ├── security_groups
        ├── services
        ├── task_definitions
        └── vpc
```
